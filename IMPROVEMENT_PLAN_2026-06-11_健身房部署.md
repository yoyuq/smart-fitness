# Smart Fitness 改进规划 — 面向健身房部署场景

> 2026-06-11 · 基于 v1.0 (tag) + multi-camera 提交 + 当前未提交改动的现状分析
> 场景定位变化：从「个人家用（一人一机一摄像头）」升级为「健身房部署（多机位、多用户、公共场所）」

---

## 一. 现状评估

### 已具备的能力（v1.0 闭环完整）
- 全链路跑通：ESP32-CAM → 后端 MediaPipe 推理 → 7 类动作分类 → 评分/计数 → WS 推送 → Android HUD → LLM 教练点评
- ESP32 固件成熟度高：动态 JPEG 质量、帧间差分跳传、指数退避重连、OTA、light sleep、服务器下发采样间隔
- 三种摄像头来源统一接入（ESP32 / Phone / PC），端侧压缩、后端统一推理
- 训练状态机 + 自动复活机制（最新未提交改动）
- 完整的账号体系、成就/PB/日历、CSV 导出、AI 计划生成

### 关键指标（实测）
- 单帧骨架推理 24ms (CPU)，并发 51 帧/s（10 并发 0 错误）
- 分类器 98% 准确率 —— **注意：在合成数据测试集上**，真实数据准确率未验证

## 二. 健身房场景与现架构的差距（按严重度排序）

### ⛔ 差距 1：多人同框 —— 当前最致命的技术缺口
- MediaPipe BlazePose 是**单人**姿态模型，默认锁定画面中"最显著"的人
- 健身房机位必然拍到路人/邻位会员 → 骨架会在人之间跳变，计数和评分直接失效
- **必须**在 pose 之前加一层：人体检测 → 目标锁定 → 裁剪主体 → 再做 pose
- 仓库里已有 `yolov8n-pose.onnx/pt`，检测能力是现成的，缺的是"锁定哪个人"的策略与跟踪

### ⛔ 差距 2：用户-机位绑定流程
- 当前绑定靠 APP 手填 device_id —— 家用可以，健身房不可行
- 需要"到站即绑"流程：每个器械区贴二维码（含 device_id），用户 APP 扫码 → 绑定该机位 → 训练 → 离开自动解绑（idle 超时已有雏形）
- 后端 `device_user_binding` 表已存在，缺的是扫码 UX + 抢占/排队逻辑（两人同时扫同一机位怎么办）

### ⛔ 差距 3：隐私合规（公共场所摄像头，PIPL 红线）
- 健身房属公共经营场所，人体图像/体态数据涉个人信息保护法的敏感个人信息范畴
- 已有的好基础：原图 24h 自动清理、device 端不存图、关键帧 30 天策略
- 缺：用户知情同意流程（注册时勾选 + 场地公示）、非会员路人入镜的处理（差距 1 的主体锁定 + 不存全帧原图，只存骨架坐标）、数据删除权接口（注销即删）

### ⚠️ 差距 4：并发与存储天花板
- 算力账：单核 ~40 帧/s，一个机位训练态 2fps → 理论 20 机位打满单进程；实测 51 req/s 含网络开销
- 中小健身房（10–20 机位）单机勉强够，但 SQLite 写并发 + 单 uvicorn 进程是瓶颈
- 路径明确：推理拆独立 worker 池（多进程/多实例）→ PostgreSQL → 按区域横向扩

### ⚠️ 差距 5：设备配网与机队管理
- WiFi SSID/密码、SERVER_HOST、TOKEN 全部硬编码在固件里 —— 部署 20 块板子要编译 20 次
- 需要：首次上电进 AP 配网模式（WiFiManager 方案）+ 配置存 NVS + 后端设备注册自发现
- OTA 已有但是单板手动推；机队需要后端统一管理（版本、在线状态、批量升级）

### ⚠️ 差距 6：模型真实性
- 分类器训练数据是 220 段**合成**数据，健身房真实光照/角度/衣着/遮挡下准确率未知
- 健身房部署反而是优势：是天然的真实数据采集场 —— 设计好"采集-标注-回流-重训"管道即可

### 🧹 差距 7：工程卫生（不影响功能但影响协作与安全）
- **敏感信息已进 git 历史**：v1.0 提交里有真实 WiFi 密码、DEVICE_TOKEN、OTA 密码（本次未提交改动已脱敏源码，但历史还在）；`_token.txt`、`cert.pem/key.pem`、`fitness.db`（15MB 含用户数据）也在仓库里
- 根目录 100+ 文件：12 个 APK（~100MB）、30+ 个 `_xxx.py` 临时脚本、20+ 个 `u*.xml/ut*.xml` 调试快照、ESP32 整个 build 产物目录（30MB+ .elf/.map）
- `quant/`、`quant_ui/`、`quant_*.py` 是无关的量化交易项目混在仓库里
- `backend/main_broken_2026-05-27.py`、`.bak` 等备份文件

## 三. 分阶段规划

### Phase 0 — 工程治理与止血（1–2 天）
1. 提交当前脱敏改动 + 自动复活功能（已在工作区）
2. **轮换全部已泄露凭据**：WiFi 密码、DEVICE_TOKEN、OTA 密码、JWT secret；新值只放 `.env` / NVS，不进 git
3. `.gitignore` 补全：`*.apk`、`edge_esp32cam/build/`、`*.db*`、`*.pem`、`_*.py`、调试图片/xml
4. 根目录清理：APK 移到 `releases/`（或 git LFS/不入库）、临时脚本归档删除、`quant*` 拆成独立仓库
5. 把 `fitness.db` 移出 git（保留迁移脚本即可）

### Phase 1 — 健身房核心能力（1–2 周）
1. **多人同框处理**（最高优先级）：
   - 推理管道改为 YOLO person 检测 → 选主体（初版策略：画面中心+最大 bbox）→ 裁剪 → MediaPipe pose
   - 加简单跟踪（IoU 匹配即可）防止帧间主体跳变
   - 验收标准：两人入镜，骨架始终锁定锻炼者，计数不受路人干扰
2. **扫码绑定机位**：器械区二维码（编码 device_id）→ APP 扫码绑定 → idle 超时自动解绑 → 机位占用状态接口（防冲突）
3. ESP32 配网改造：AP 配置门户 + NVS 存储，固件一次编译全场通用
4. MJPEG/WS 断线重连补全（IMPROVEMENT_PLAN_2026-05-28 中已识别，未做）

### Phase 2 — 数据与模型升级（2–4 周）
1. 真实数据回流管道：训练 session 的骨架序列（不存原图）自动归集 → 标注工具（复用现有 session 回放）→ 周期性重训分类器
2. 评分规则 YAML 化（沿用 05-28 规划的方案 B）：教练可自定义动作规则，不改代码加新动作
3. rep 计数时序平滑：滑动窗口 + 状态机去抖，解决真实场景下的误计/漏计
4. 用真实数据重测准确率，建立真实测试集基准（替代合成 98% 这个数字）

### Phase 3 — 规模化与运维（1–2 月）
1. SQLite → PostgreSQL（迁移脚本基础已有 migrate_v3.py 模式）
2. 推理服务拆独立 worker 池：uvicorn 多 worker 或独立推理进程 + 队列，按机位数水平扩
3. 设备机队管理后台：在线状态、固件版本、批量 OTA、每机位帧率/延迟监控
4. 隐私合规包：注册同意书 + 数据删除接口 + 留存策略自动执行审计 + 场地公示文案
5. 健身房运营侧报表：机位利用率、会员活跃度、教练端查看会员动作质量趋势

### Phase 4 — 远期增值（按需）
- 边缘推理下沉：每区域一台小主机（NUC 级）跑检测+pose，ESP32 只推流到局域网，降低中心负载与隐私风险
- 机位端语音教练（站内音箱/手环震动反馈）
- 教练端 APP/网页：实时查看多会员训练状态
- 时序模型（LSTM/TCN）做节奏与离心控制分析（05-28 方案 C，数据积累够了再做）

## 五. 算力规划速查

| 机位数 | 训练态请求量 | 单机单进程 (51 req/s) | 建议 |
|--------|--------------|----------------------|------|
| ≤10    | ≤20 req/s    | ✅ 富余               | 现架构直接用 |
| 10–20  | 20–40 req/s  | ⚠️ 接近上限           | uvicorn 多 worker |
| 20–50  | 40–100 req/s | ❌ 超限               | 推理 worker 池 + PG |
| >50    | >100 req/s   | ❌                   | 分区域边缘推理下沉 |

（按训练态 2fps 计；加多人检测层后单帧成本约 +15–30ms，上表需打 7 折）
